1️⃣ Preparación

¿Qué es? La preparación es la fase más importante y la más descuidada. Es todo lo que haces antes de que ocurra un incidente para estar listo cuando ocurra. Como tener un extintor en casa antes de que haya un incendio.

¿Por qué es importante? "La mejor manera de responder a un incidente es estar preparado para él." - NIST SP 800-61

Una buena preparación puede:

  • Reducir el tiempo de respuesta
  • Minimizar el daño
  • Ahorrar dinero
  • Proteger la reputación de la organización
Ejemplo de plan

2️⃣ Detección y análisis

¿Qué es? La detección es descubrir que está ocurriendo un incidente. Es como el sistema de alarma de tu casa que te avisa cuando hay un intruso.

Una detección temprana:

  • ⏱️ Reduce el daño
  • 🔍 Permite una contención más efectiva
  • 💰 Ahorra dinero y recursos

Indicadores de incidente

  • Escaneo de puertos: alguien sondeando tu servidor.
  • Pruebas de vulnerabilidades: buscando fallos.
  • Falsas autenticaciones: intentos de login fallidos.
  • Descargas sospechosas: archivos extraños.
  • Alertas de antivirus: malware detectado.
  • Archivos sospechosos: nombres extraños.
  • Cambios inesperados: modificaciones sin autorización.
  • Rendimiento anormal: servidor lento.
  • Actividad de red extraña: tráfico a IPs desconocidas.
  • Logs inusuales: entradas anormales.
  • Reportes de usuarios: usuarios detectan anomalías.
Detección de Web Shell
# Buscar archivos PHP con funciones peligrosas
# Esto identifica scripts sospechosos que pueden ejecutar comandos
grep -r "system\|eval\|shell_exec\|exec\|passthru" /var/www/html/uploads/

# Buscar archivos recién modificados
# Un web shell suele aparecer con fecha reciente tras una intrusión
find /var/www/html/uploads/ -type f -mtime -1

# Buscar archivos con nombres sospechosos
# Se buscan extensiones ejecutables dentro de uploads
find /var/www/html/uploads/ -name "*.php" -o -name "*.jsp" -o -name "*.aspx"

# Revisar logs de acceso
# Buscar solicitudes a archivos subidos que podrían ser un shell
grep "uploads/.*\.php" /var/log/apache2/access.log
grep "cmd\|exec\|system" /var/log/apache2/access.log
grep "GET /uploads/.*\.php" /var/log/apache2/access.log | awk '{print $1}' | sort | uniq -c

# Revisar procesos en ejecución
# Verificar si hay procesos inusuales o sospechosos
ps aux --sort=-%cpu | head -10
ps aux | grep -v "root\|www-data\|mysql\|apache"
3️⃣ Contención

¿Qué es? Detener el daño. Es como cuando los bomberos crean un perímetro para que el fuego no se propague.

🛑Regla de oro: No elimines nada durante la contención. Primero guarda evidencia.

Estrategias

  • Corto plazo: desconectar sistemas, apagar servicios, bloquear IPs, aislar el sistema.
  • Largo plazo: parchear, cambiar contraseñas, revisar configuraciones y mantener aislamiento.
Comandos de contención
# 1. Desconectar del servidor (¡con cuidado si es remoto!)
# Esto aísla el sistema para evitar más daño
ifconfig eth0 down

# 2. Añadir reglas de firewall restrictivas
# Bloquea todo tráfico excepto localhost y la red de confianza
iptables -P INPUT DROP
iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -s 192.168.1.0/24 -j ACCEPT

# 3. Apagar servicios
# Detiene servicios potencialmente comprometidos para contener el incidente
systemctl stop apache2
systemctl stop mysql
systemctl stop nginx

# 4. Bloquear IPs maliciosas específicas
iptables -A INPUT -s 45.33.22.11 -j DROP
iptables -A INPUT -s 185.130.5.253 -j DROP

Preservación de evidencia

  • Archivos maliciosos: copiar a un disco externo, no eliminar.
  • Logs: copiar a un lugar seguro, no modificar.
  • Capturas de pantalla: tomar fotos.
  • Memoria: usar dd para copiar RAM.
  • Tráfico: guardar capturas pcap.
Preservación de evidencia
# Copiar archivos maliciosos sin ejecutarlos
# Guarda los archivos sospechosos para análisis forense
cp -r /var/www/html/uploads/ /evidencia/uploads_backup/

# Copiar logs
# Conserva registros de actividad sin alterarlos
cp /var/log/apache2/access.log /evidencia/logs/access.log
cp /var/log/apache2/error.log /evidencia/logs/error.log
cp /var/log/syslog /evidencia/logs/syslog

# Crear hash de archivos para integridad
# Permite demostrar que los archivos no fueron modificados después
sha256sum /var/www/html/uploads/shell.php > /evidencia/shell.php.sha256

# Copiar memoria
# Volcar la memoria RAM para analizar procesos y artefactos en vivo
sudo dd if=/dev/mem of=/evidencia/memory.dump bs=1M count=1024

# Capturar tráfico de red
# Guardar un archivo pcap con el tráfico relevante del incidente
tcpdump -i eth0 -w /evidencia/traffic.pcap
4️⃣ Erradicación

¿Qué es? Eliminar la causa raíz del incidente. Es como reparar la falla que causó el fuego.

Acciones clave

  • Eliminar malware
  • Verificar puertas traseras
  • Actualizar el código vulnerable
  • Cambiar contraseñas comprometidas
Buscar y eliminar malware
# Buscar archivos sospechosos por contenido
# Identifica scripts que contienen funciones peligrosas frecuentemente usadas por malware
grep -r "eval\|system\|shell_exec\|passthru\|curl_exec" /var/www/html/

# Buscar archivos cambiados recientemente
# Un cambio reciente puede indicar una implantación maliciosa
find /var/www/html/ -type f -mtime -7 -exec ls -la {} \;

# Buscar archivos con permisos extraños
# Permisos 777 son un indicador de riesgo
find /var/www/html/ -type f -perm -777 -exec ls -la {} \;

# Buscar archivos ocultos
# Los archivos empezando con . pueden ser backdoors ocultos
find /var/www/html/ -type f -name ".*" -exec ls -la {} \;

# Eliminar archivos sospechosos
rm -f /var/www/html/uploads/shell.php
rm -f /var/www/html/uploads/backdoor.php
rm -rf /tmp/malware_folder/

# Eliminar tareas cron maliciosas
# Revisa si hay trabajos programados que reinicien el malware
crontab -e
Contraseñas seguras
# Generar contraseña segura en Linux
# openssl usa una fuente de entropía para crear una cadena aleatoria
openssl rand -base64 32

# Generar contraseña segura en Python
# secrets.token_urlsafe ofrece un valor seguro para contraseñas y tokens
python3 -c "import secrets; print(secrets.token_urlsafe(32))"

# Generar contraseña segura en línea de comandos
# Usa la hora actual y un hash para producir una cadena difícil de adivinar
date +%s | sha256sum | base64 | head -c 32 ; echo
5️⃣ Recuperación

¿Qué es? Volver a la normalidad después del incidente. Es como reparar la casa y volver a vivir en ella.

Restauración
# 1. Verificar que la copia de seguridad es limpia
# Confirma que el backup no contiene malware ni archivos alterados
# 2. Detener servicios
systemctl stop apache2
systemctl stop mysql

# 3. Restaurar archivos
rsync -avz /backup/ultimo_limpio/ /var/www/html/

# 4. Restaurar base de datos
# Restaura la base de datos desde una copia de seguridad confiable
mysql -u root -p database < /backup/database_limpio.sql

# 5. Iniciar servicios
systemctl start apache2
systemctl start mysql

# 6. Verificar que todo funciona
curl -I https://tudominio.com
# Comprueba que el servidor responde y que el certificado funciona correctamente

Pruebas clave

  • Abrir página principal
  • Subir un archivo legítimo
  • Iniciar sesión
  • Consultar base de datos
  • Intentar subir shell.php
  • Comprobar rendimiento

Monitoreo

  • Logs de acceso
  • Rendimiento
  • Archivos nuevos
  • Tráfico de red
  • Alertas de seguridad
6️⃣ Lecciones aprendidas

¿Qué es? Analizar qué pasó, por qué pasó y cómo evitar que vuelva a pasar. Es la fase más importante para la mejora continua.

¿Por qué es tan importante? "Aquellos que no recuerdan el pasado están condenados a repetirlo." - George Santayana

Reunión post-mortem

  • Equipo de respuesta
  • Desarrolladores
  • Administradores de sistemas
  • Representantes del negocio
  • Expertos externos (opcional)

Preguntas clave

  • ¿Qué pasó exactamente?
  • ¿Cuándo ocurrió?
  • ¿Cómo ocurrió?
  • ¿Por qué ocurrió?
  • ¿Qué funcionó bien?
  • ¿Qué no funcionó?
