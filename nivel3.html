¿Cómo se habría evitado?

Shift Left Security · Prevenir desde el inicio

5 preguntas Mínimo 7/10 para aprobar

El Arte de la Prevención: La única forma real de evitar estos problemas es pensar en seguridad desde el principio. No es algo que se añade al final, sino la base sobre la que construyes.

Imagina que estás construyendo una casa. ¿Pondrías las cerraduras de las puertas después de que los ladrones ya hayan entrado? No. Las pones en el plano, desde el principio. Eso es Shift Left Security: mover la seguridad hacia la izquierda, al inicio del proceso de desarrollo.


1. Fase de análisis y diseño

Política de archivos: define con claridad qué tipo de archivos se aceptan, por ejemplo solo .pdf, .docx o .jpg, y cuál es su tamaño máximo.

  • ✅ Almacenamiento seguro: guarda los archivos fuera de la carpeta pública del sitio web para que, si se sube algo malicioso, no pueda ejecutarse desde el navegador.
  • ✅ Nombres de archivo: nunca uses el nombre original. Genera un nombre aleatorio e irrepetible, como un UUID, para guardar el archivo en el servidor.
2. Diseño seguro
// Cada usuario tiene su propia carpeta
/uploads/estudiante1/
// ruta dedicada para el estudiante 1
/uploads/estudiante2/
// ruta dedicada para el estudiante 2
// Nombres de archivo generados automáticamente
123e4567-e89b-12d3-a456-426614174000_tarea.pdf
// un nombre único evita colisiones y oculta el nombre original

Este diseño evita que un atacante adivine rutas o acceda a archivos que no le pertenecen.

3. Desarrollo (código seguro)

Las validaciones que hace el navegador son útiles para la experiencia del usuario, pero no son seguras. Un atacante puede saltárselas fácilmente. Por eso, toda validación importante debe hacerse en el servidor.

  • ✅ Validación del tipo (MIME y cabecera): verifica el tipo de archivo que dice ser el navegador, aunque esto no es suficiente porque se puede falsear.
  • ✅ Validación del tipo (firma del archivo): la mejor técnica es leer los primeros bytes para confirmar que el archivo es lo que dice ser.
  • ✅ Validación del tamaño: limita el tamaño máximo para evitar ataques de denegación de servicio.
  • ✅ Sanitización del nombre: si necesitas usar el nombre original, elimina caracteres peligrosos como .., /, \ o ;.
// Código mejorado paso a paso
@PostMapping
// mapea la petición HTTP POST al método subirArchivo
public ResponseEntity<?> subirArchivo(
// declara el método que devuelve una respuesta HTTP genérica
    @RequestParam("file") MultipartFile archivo,
// recibe el archivo enviado desde el formulario
    Authentication autenticacion) {
// recibe también los datos del usuario autenticado

    // 1. Verificar quién es el usuario
    String usuario = autenticacion.getName();
// obtiene el nombre o identificador del usuario conectado

    // 2. Verificar el tamaño
    if (archivo.getSize() > 10_000_000) {
        return ResponseEntity.badRequest()
            .body("❌ Archivo demasiado grande");
    }
// rechaza archivos mayores a 10 MB para evitar abuso

    // 3. Verificar el tipo de archivo
    String tipo = archivo.getContentType();
// obtiene el tipo MIME declarado por el navegador
    if (!TIPOS_PERMITIDOS.contains(tipo)) {
        return ResponseEntity.badRequest()
            .body("❌ Tipo no permitido");
    }
// rechaza tipos de archivo que no están en la lista blanca

    // 4. Guardar con nombre seguro
    String nombreSeguro = UUID.randomUUID().toString() + "_" + limpiarNombre(...);
// genera un nombre único para evitar colisiones y ataques de path traversal
    Path destino = Paths.get("/uploads/seguro/" + usuario, nombreSeguro);
// guarda el archivo dentro de una carpeta segura por usuario
    Files.write(destino, archivo.getBytes());
// escribe el contenido real del archivo en el disco

    return ResponseEntity.ok().body("✅ Archivo subido");
// devuelve una respuesta exitosa al cliente
}
4. Fase de pruebas (QA)

Pruebas negativas: no solo verifiques que subir un PDF funciona. Prueba también casos adversos como archivos .php, archivos enormes, nombres como ..//..//config.php y valida que el sistema los rechace.

  • ✅ Subir un PDF de 2 MB → Debe funcionar
  • ❌ Subir un archivo de 50 MB → Debe ser rechazado
  • ❌ Subir un .exe → Debe ser rechazado
  • ❌ Subir ../../etc/passwd → Debe ser rechazado
